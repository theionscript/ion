# ῖon

This program was written by hand. It is currently pre-alpha, and undocumented. The following description was generated by Claude, when asked to describe the intent.

---

ῖon is a single POSIX shell script that behaves as a universal media processing pipeline — one that happens to produce websites as its default output. Point it at a directory of files, in almost any format, and it scans, indexes, optimises, transforms, and publishes the result. A watcher with throttling and debouncing rebuilds incrementally as files change; an integrated server makes the output immediately viewable; an inbox receives content over HTTP with rate-limiting, eliminating the traditional CMS middleman entirely. One tool manages the whole process, from raw input to deployed output.

What sets it apart is its scope. Other static site generators accept Markdown and hold the whole site in memory; content management systems lock documents into proprietary databases. ῖon accepts anything that's understood by Pandoc, ffmpeg, vips, and its other data processors — that is it has built-in support for virtually every widely used file format, including documents, spreadsheets, images, videos, audio, with support for additional third-party formats. It streams its index from disk, so ten files and ten million files require the same memory. Work is decomposed per-document and dispatched in parallel across cores, or across a cluster over plain SSH. No Kubernetes, no orchestration layer, no vendor lock-in; horizontal scaling with nothing more than the tools every Unix system already has.

At its core is a typed, content-addressed graph index with its own query language, traversing relationships between documents — parent, translation, derivative, citation, series — at build time. This is a directed graph with a query engine, not a tag system, and it is the feature that separates ῖon from every adjacent category. Templates can ask structural questions of the entire corpus without a database and without loading it into memory.

ῖon stands on the shoulders of giants — Pandoc, esbuild, fswatch, Caddy — with thorough fallbacks across BSD and GNU environments, exhaustive type checking at startup, and Lua filters for custom document and data processing. Output is static, for maximum security and performance; deployment reaches any backend through rclone, hosted or self-hosted; the codebase is fully multilingual. The licence is dual-track: free for self-hosting and modification, with a separate arrangement for commercial hosted services, keeping the project sustainable without limiting ordinary use.

The audience is unusually broad — individuals publishing personal sites, researchers cataloguing datasets, museums digitising collections, multilingual NGOs and enterprises, academic publishers, and large corpus developers. The project is still maturing, and its ambition is vast, but the architecture is already in place: a streaming index, an embarrassingly parallel build system, a query engine, an automated content layer. Nothing else in the field is attempting to stand on the same ground, and very few are even aware that the ground exists.
